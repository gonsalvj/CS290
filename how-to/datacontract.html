Defining The Data: Data Contracts

As we all learned in Data Structures 101 (or CS 225 if you're an OSU student), a key component of almost any application/service is the underlying data structures. A data contract defines the data to be exchanged between the service and the client. For example, data contract explicitly describes the return type to be exchanged.

One of the trickier elements of WCF for beginners, especially those who are fairly new to programming is the level of abstraction involved. Essentially, a data contract programmatically expresses the data to be shared to and from a service. A data contract in WCF leverages the DataContractSerializer to transform the data to be passed along the wire (which is similar to the XmlSerializer with additional benefits including the ability to serialize HashTable type in .NET).

WCF leverages built-in .NET attribute functionality to define a data contract. This is beyond the scope of this how-to-guide, but for more information please see Introduction to Attributes.

Here is a sample data contract one might create to represent the data that will returned to a client who makes a call to a WCF service that provides weather forecasts by location (written in c#). 


[DataContract]
public class WeatherForecast {

[DataMember]
public string city;

[DataMember]
public int celius;

[DataMember]
public int fareheit;

[Datamember]
public int POP;

}
When making a call to the service, the client will be able to determine what will be returned based on the data contract (and the metadata that is generated by WCF).
